## Adding or updating strings

1. Use WordPress internationalization functions for all user-facing PHP strings.
2. Use the plugin text domain exactly as written above.
3. Update the POT file from the current source.
4. Update only the affected reviewed PO files.
5. Preserve placeholders, HTML tags and formatting tokens exactly.
6. Compile the MO files.
7. Validate the PO catalogs and test both the site locale and the user-profile locale.
8. Test `de_DE` with the informal “du” form, `de_DE_formal` with the formal “Sie” form, `es_ES` entirely in Spanish, `fr_FR` entirely in French, `it_IT` entirely in Italian, `hi_IN` entirely in Hindi, `ru_RU` entirely in Russian, `pt_BR` entirely in Brazilian Portuguese, `zh_CN` entirely in Simplified Chinese, `ja` entirely in Japanese, `nl_NL` entirely in Dutch and `pl_PL` entirely in Polish, `tr_TR` entirely in Turkish and `fa_IR` entirely in Persian, `vi` entirely in Vietnamese and `cs_CZ` entirely in Czech. Do not mix languages or forms of address.

## WordPress.org language packs

WordPress.org language packs may take precedence over bundled translations. Translation imports should be submitted only when every string is consistently written in the target language. Machine-generated catalogs require complete human review before submission.
